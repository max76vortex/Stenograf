# Technical Specification

**MAS Project ID:** `audio-transcription-v1`  
**Document version:** `1.2-delta`  
**Mode:** DELTA v1.2 over current run (no new MAS run)

## 1. General Description and Goal

- Реализовать и довести до практического использования локальный конвейер транскрибации накопленных аудиозаписей (`~30 GB`, преимущественно `mp3`) в заметки Obsidian vault `Audio Brain`.
- Целевой результат: в `00_inbox` создаются `.md`-транскрипты с предсказуемым именованием и однозначной связью с источником через поле `audio_file`.
- Обработка должна быть повторно запускаемой без перезаписи готовых результатов по умолчанию.
- Дельта v1.2: продолжить текущий run `audio-transcription-v1`, разделив delivery и R&D на параллельные потоки, при этом Core-поток должен принимать ASR-изменения только через provider abstraction и подтвержденный benchmark decision.

## 2. Area of Changes in Existing System

- Контур изменений: `transcription/` и run-артефакты `multi-agent-system/current-run/`.
- Входит в область:
  - сценарии транскрибации в `transcribe_to_obsidian.py` (включая `--recursive`, `--manifest`);
  - проверка покрытия `check_coverage.py`;
  - эксплуатационная документация (`README.md`, `SETUP.md`, примеры запуска);
  - контракт работы Core/MAS Delivery с ASR R&D результатами.
- Не входит в область:
  - интеграции с Ghost/CMS;
  - n8n-потоки публикации;
  - ручная/LLM-редактура контента как часть этого прогона.

## 3. Use Cases

### UC-01: Установка окружения для локальной транскрибации

- **Actors:** пользователь, локальная система Python/CUDA.
- **Preconditions:** Windows-машина; доступ на установку Python и зависимостей; интернет для загрузки модели.
- **Main scenario:**
  1. Пользователь устанавливает Python 3.10+.
  2. Создает и активирует virtual environment.
  3. Устанавливает зависимости из `transcription/requirements.txt`.
  4. Выполняет проверочный запуск CLI.
- **Alternative scenarios:**
  - A1: CUDA недоступна -> используется CPU-режим.
  - A2: первая загрузка модели занимает дополнительное время и место на диске.
- **Postconditions:** инструменты запускаются локально без ошибок импорта.
- **Acceptance criteria:**
  - [ ] `SETUP.md` достаточно для воспроизведения установки на новой машине.
  - [ ] В документации есть проверочные команды (как минимум версия Python и тестовый `--help`).

### UC-02: Транскрибация папки месяца в `00_inbox`

- **Actors:** пользователь, `transcribe_to_obsidian.py`.
- **Preconditions:** входная папка содержит `mp3`; выходная папка `Audio Brain/00_inbox` существует.
- **Main scenario:**
  1. Пользователь запускает транскрибацию для одной папки месяца.
  2. Для каждого не обработанного файла создается `.md`-заметка.
  3. В заметке присутствует frontmatter и блок "Транскрипт".
- **Alternative scenarios:**
  - A1: соответствующий `.md` уже есть -> файл пропускается.
  - A2: включен `--overwrite` -> существующий результат перезаписывается.
  - A3: включен `--manifest` -> фиксируется журнал обработки.
- **Postconditions:** в `00_inbox` появляются новые транскрипты, каждый связан с исходным аудио.
- **Acceptance criteria:**
  - [ ] В каждой заметке заполнено поле `audio_file` с именем исходника.
  - [ ] Именование выходного `.md` детерминировано и не конфликтует при корректных входных именах.

### UC-03: Рекурсивная транскрибация дерева `recordings/`

- **Actors:** пользователь, `transcribe_to_obsidian.py` с `--recursive`.
- **Preconditions:** структура входа соответствует `recordings/YYYY-MM/`.
- **Main scenario:**
  1. Пользователь запускает один рекурсивный прогон от корня `recordings`.
  2. Скрипт обрабатывает подпапки и пишет результаты в плоский `00_inbox`.
- **Alternative scenarios:**
  - A1: файлы с одинаковым stem в разных подпапках требуют уникального результата без взаимной перезаписи.
  - A2: часть результатов уже существует -> обрабатываются только отсутствующие.
- **Postconditions:** готов полный набор новых транскриптов для необработанных аудио в дереве.
- **Acceptance criteria:**
  - [ ] Для одинаковых stem из разных подпапок не возникает перезаписи одного результата другим.

### UC-04: Проверка покрытия транскрибации

- **Actors:** пользователь, `check_coverage.py`.
- **Preconditions:** доступны пути к source `recordings` и destination `00_inbox`.
- **Main scenario:**
  1. Пользователь запускает проверку покрытия.
  2. Система показывает список аудио без ожидаемого `.md`.
- **Alternative scenarios:**
  - A1: при полном покрытии список отсутствующих файлов пуст.
  - A2: используется `--recursive` для проверки всего дерева.
- **Postconditions:** пользователь получает точный список "что еще не транскрибировано".
- **Acceptance criteria:**
  - [ ] Правило сопоставления "аудио -> ожидаемый `.md`" синхронизировано с логикой транскрибации и задокументировано.

### UC-05: Повторный запуск после прерывания на большом объеме

- **Actors:** пользователь, локальная вычислительная среда (GPU/CPU).
- **Preconditions:** долгий прогон может быть остановлен/прерван.
- **Main scenario:**
  1. Пользователь запускает обработку большого набора данных.
  2. При прерывании выполняет повторный запуск тем же способом.
  3. Система продолжает с необработанных файлов.
- **Alternative scenarios:**
  - A1: при необходимости пользователь включает `--overwrite` для полного перегенерирования.
- **Postconditions:** восстановление выполнения возможно без потери уже готовых результатов.
- **Acceptance criteria:**
  - [ ] По умолчанию повторный запуск не перезаписывает существующие транскрипты.

### UC-06: Delta-контур Core/MAS Delivery и ASR R&D

- **Actors:** Worktree 1 (Core/MAS Delivery), Worktree 2 (ASR R&D), пользователь как утверждающая сторона.
- **Preconditions:** run `audio-transcription-v1` уже существует и продолжается без перезапуска.
- **Main scenario:**
  1. Core-поток развивает рабочий конвейер транскрибации.
  2. R&D-поток выполняет benchmark/валидацию ASR и формирует decision package.
  3. Core принимает изменения по ASR только через согласованный интерфейс провайдера и актуальный decision package.
- **Alternative scenarios:**
  - A1: если decision package не обновлен или не подтвержден, Core не меняет ASR-провайдер.
- **Postconditions:** параллельная работа потоков не ломает основной конвейер и не создает невалидированных ASR-изменений.
- **Acceptance criteria:**
  - [ ] Текущий run явно помечен как delta v1.2 continuation.
  - [ ] В Core отсутствует жесткая привязка к единственному ASR backend.
  - [ ] Решения по ASR в Core опираются на актуальный benchmark decision из R&D.

## 4. Non-Functional Requirements

- Локальная обработка аудио без отправки данных во внешние сервисы в рамках базового конвейера.
- Совместимость с Windows 10/11 и Python 3.10+.
- Поддержка GPU-режима (`cuda`) при наличии совместимого оборудования; CPU-режим допустим как fallback.
- Идемпотентность запуска (без `--overwrite`) для долгих и повторных прогонов.
- Предсказуемость структуры выхода: единый формат заметки, стабильное именование и связь с исходником.

## 5. Constraints and Assumptions

- Базовый входной формат текущего прогона: `mp3`.
- Структура входных данных: `recordings/YYYY-MM/` с шаблоном имени `YYYY-MM-DD_NNN[_метка].mp3`.
- Выходной vault: `D:\Obsidian\Audio Brain\00_inbox\` (путь задается пользователем, не захардкоживается).
- Модель `large-v3` загружается при первом запуске и требует дискового места под кэш.
- Постинг в Ghost/CMS вынесен за рамки данного прогона.
