# TZ Review - Dry Run: Service status surface

## 1) Verdict

**APPROVED**

## 2) Что проверено

- Соответствие постановке: наличие единой точки входа, отображение статуса, блокирующих вопросов и артефактов активного прогона, работа внутри контура `multi-agent-system/` без тяжелой инфраструктуры.
- Согласованность с ограничениями проекта: без новых внешних сервисов и БД, без ломания n8n, минимальные зависимости, допустимость страницы или CLI.
- Полнота функциональных и нефункциональных требований относительно dry run (читаемый вывод, устойчивость к отсутствующим файлам).
- Учет рисков (форматы markdown, неоднозначность "последних" артефактов) и наличие плана смягчения (толерантный парсинг, детерминированное правило отбора на этапе планирования).
- Ограничение по Cursor CLI в `PATH` отражено и не конфликтует с выбранным минимальным стеком.

## 3) Замечания

### Critical

- Нет.

### Major

- **FR3 / формулировка "recent task/review/report artifacts"** может расходиться с постановкой про активный прогон (`current-run/`). На этапе Architecture нужно явно зафиксировать: перечень артефактов только под `multi-agent-system/current-run/` (и правила вложенных путей), либо однозначно описать исключения, если "recent" подразумевает что-то вне текущего run.
- **FR1 (user-confirmed stages list)** расширяет исходный brief; это обосновано для dry run, но источник истины для этого списка в ТЗ не назван (отдельный файл, секция в `status.md` и т.д.). Стоит зафиксировать в архитектурном шаге, иначе реализация может уйти в эвристики.

### Minor

- Уточнить в реализации/архитектуре единый контракт полей для `status.md` (минимальный набор ключей/секций), чтобы "толерантный парсинг" не превратился в нестабильный вывод между прогонами.
- Если точка входа - "страница в существующем контексте проекта", заранее определить, как не затронуть сборку/деплой n8n (граница модулей, статическая страница, изоляция маршрута).

## 4) Blocking Questions

**None**

_(Для этапа Architecture полезно согласовать не блокирующее, но важное уточнение: единственный ли корень артефактов - `multi-agent-system/current-run/`, включать ли подкаталоги и по какому правилу сортировать "последние" файлы.)_

## 5) Рекомендации к следующему этапу (Architecture)

- Зафиксировать **детерминированное правило "latest artifacts"**: критерий (mtime vs явный индекс/манифест), глубина обхода `current-run/`, фильтры по расширениям и исключениям (например, служебные файлы).
- Описать **схему извлечения данных** из `status.md` и `open_questions.md` (допустимые варианты разметки, fallback при пустых секциях) и где хранится **список подтвержденных стадий**, если он обязателен для приемки.
- Выбрать и обосновать **одну точку входа** (скрипт на stdlib/stack репозитория vs минимальная статическая страница), с явным требованием "без новых тяжелых зависимостей" и без изменения поведения существующих workflow n8n.
- Заложить **smoke-проверку dry run**: сценарий с отсутствующими опциональными файлами и с частично заполненным `status.md`, ожидаемый человекочитаемый вывод.
