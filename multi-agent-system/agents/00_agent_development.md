# Мультиагентная система разработки

## Назначение

Эта система нужна, чтобы разбивать разработку на роли с узким контекстом и
собирать итоговый результат через оркестратор, а не пытаться делать весь цикл
в одном диалоге и в одном контекстном окне.

## Роли

### Оркестратор

- ведет последовательность стадий;
- передает входы между агентами;
- обновляет `multi-agent-system/status.md`;
- останавливает процесс на блокирующих вопросах;
- отдает результат на подтверждение пользователю после каждого этапа.

### Аналитик

- делает ТЗ;
- формирует use case;
- фиксирует критерии приемки;
- выносит открытые вопросы.

### Ревьюер ТЗ

- проверяет полноту ТЗ;
- ловит пробелы, противоречия и неконкретные критерии;
- не переписывает ТЗ вместо аналитика.

### Архитектор

- превращает ТЗ в прикладную архитектуру;
- описывает компоненты, интерфейсы, модель данных и ограничения;
- держит архитектуру реализуемой и простой.

### Ревьюер архитектуры

- проверяет выполнимость архитектуры;
- особенно тщательно проверяет модель данных, интеграции и совместимость.

### Планировщик

- делает детальный план задач сверху вниз;
- сначала каркас и заглушки, затем полноценная реализация;
- создает task-файлы, по которым разработчик может работать без домысливания.

### Ревьюер плана

- формально проверяет покрытие use case;
- проверяет наличие task-файлов и обязательных секций;
- не подменяет собой архитектора или разработчика.

### Разработчик

- реализует только то, что указано в задаче;
- пишет тесты и обновляет документацию;
- предоставляет отчет о тестировании;
- не рефакторит без прямого указания.

### Ревьюер кода

- сверяет код с task-файлом;
- проверяет тестовый отчет и регрессии;
- классифицирует замечания по критичности.

## Артефакты процесса

Активный прогон задачи идет в `multi-agent-system/current-run/`.

Обязательные артефакты:

- `technical_specification.md`
- `tz_review.md`
- `architecture.md`
- `architecture_review.md`
- `plan.md`
- `plan_review.md`
- `tasks/*.md`
- `reviews/*.md`
- `reports/*.md`
- `open_questions.md`

Системный статус процесса хранится в `multi-agent-system/status.md`.

## Принцип реализации

### Подход сверху вниз

1. Сначала создается каркас решения.
2. Затем формируется архитектурный и задачный скелет.
3. Потом разработчик делает минимально работающую реализацию по task-файлам.
4. Затем система усиливается тестами и деталями реализации.

### Управление неопределенностью

- Любая критичная неясность уходит в `open_questions.md`.
- Оркестратор обязан остановить процесс, если без ответа пользователя нельзя
  надежно двигаться дальше.

### Подтверждение после каждого этапа

Система в этом workspace настроена на вариант B:

- после каждого этапа есть отдельный артефакт;
- пользователь подтверждает завершение этапа;
- только после подтверждения двигаемся дальше по цепочке.
