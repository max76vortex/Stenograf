# End-To-End Flow

## Цель

Этот документ показывает, как система проходит одну задачу от постановки до
приемки результата.

## Сценарий

### Шаг 1. Оркестратор стартует задачу

Вход:

- постановка задачи;
- описание проекта;
- `multi-agent-system/agents/01_orchestrator.md`.

Выход:

- обновлен `multi-agent-system/status.md`;
- определен этап `Analysis`.

### Шаг 2. Аналитик делает ТЗ

Вход:

- постановка задачи;
- описание проекта;
- `multi-agent-system/agents/02_analyst_prompt.md`.

Выход:

- `current-run/technical_specification.md`.

Подтверждение пользователя:

- ТЗ приемлемо как база для архитектуры.

### Шаг 3. Ревьюер ТЗ проверяет ТЗ

Выход:

- `current-run/tz_review.md`.

Подтверждение пользователя:

- замечания понятны и достаточны.

### Шаг 4. Архитектор делает архитектуру

Выход:

- `current-run/architecture.md`.

Подтверждение пользователя:

- архитектура выглядит реализуемой.

### Шаг 5. Ревьюер архитектуры проверяет архитектуру

Выход:

- `current-run/architecture_review.md`.

Подтверждение пользователя:

- замечания понятны и не потеряны критичные риски.

### Шаг 6. Планировщик строит план

Выход:

- `current-run/plan.md`;
- task-файлы в `current-run/tasks/`.

Подтверждение пользователя:

- задачи достаточно детальны.

### Шаг 7. Ревьюер плана проверяет покрытие

Выход:

- `current-run/plan_review.md`.

Подтверждение пользователя:

- план полностью покрывает задачу.

### Шаг 8. Разработчик выполняет первую задачу

Выход:

- изменения в коде;
- тесты;
- отчет в `current-run/reports/`.

Подтверждение пользователя:

- формат результата и отчета устраивает.

### Шаг 9. Ревьюер кода проводит проверку

Выход:

- структурированное ревью;
- решение: approve или rework.

Подтверждение пользователя:

- решение по задаче можно принять.

### Шаг 10. Оркестратор двигается к следующей задаче или завершает прогон

Выход:

- обновленный `status.md`;
- итоговая сводка по задаче.

## Блокирующий сценарий

Если любой агент не может надежно продолжать:

1. пишет вопрос в `current-run/open_questions.md`;
2. оркестратор переводит `status.md` в `BLOCKED`;
3. пользователь отвечает;
4. процесс возобновляется с остановленного этапа.

## Минимальный критерий готовности системы

Система считается рабочей в варианте B, если:

- у каждого этапа есть свой prompt и артефакт;
- существует единая рабочая зона `current-run/`;
- оркестратор умеет остановиться на подтверждение;
- оператору достаточно подставить постановку задачи и идти по runbook’у.
