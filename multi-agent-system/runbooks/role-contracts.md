# Role Contracts

Этот документ фиксирует входы, выходы и критерии приемки по каждому агенту.

## Оркестратор

**Входы**
- постановка задачи;
- описание проекта;
- правила процесса из `multi-agent-system/agents/01_orchestrator.md`.

**Выходы**
- обновленный `multi-agent-system/status.md`;
- передача контекста следующему агенту;
- остановка на пользовательское подтверждение или на блокерах.

**Критерии приемки**
- ведет стадии без потери состояния;
- обновляет статус после каждого шага;
- не перескакивает через обязательные этапы;
- останавливает процесс по правилам.

## Аналитик

**Входы**
- постановка задачи;
- описание проекта;
- замечания ревьюера при повторной итерации.

**Выходы**
- `multi-agent-system/current-run/technical_specification.md`.

**Критерии приемки**
- есть use case;
- есть критерии приемки;
- есть открытые вопросы;
- ТЗ достаточно конкретно для архитектуры.

## Ревьюер ТЗ

**Входы**
- `technical_specification.md`;
- исходная постановка;
- описание проекта.

**Выходы**
- `multi-agent-system/current-run/tz_review.md`.

**Критерии приемки**
- замечания разделены на `BLOCKING`, `MAJOR`, `MINOR`;
- указаны проблемные места и рекомендации;
- review не переписывает ТЗ вместо аналитика.

## Архитектор

**Входы**
- ТЗ;
- описание проекта;
- замечания ревьюера архитектуры при повторной итерации.

**Выходы**
- `multi-agent-system/current-run/architecture.md`.

**Критерии приемки**
- есть компоненты, интерфейсы, модель данных;
- архитектура реализуема;
- учтены ограничения проекта;
- хватает деталей для планирования.

## Ревьюер архитектуры

**Входы**
- `architecture.md`;
- ТЗ;
- описание проекта.

**Выходы**
- `multi-agent-system/current-run/architecture_review.md`.

**Критерии приемки**
- проверены выполнимость и совместимость;
- проверена модель данных;
- замечания конкретны и ранжированы по критичности.

## Планировщик

**Входы**
- ТЗ;
- архитектура;
- описание проекта;
- код проекта.

**Выходы**
- `multi-agent-system/current-run/plan.md`;
- task-файлы в `multi-agent-system/current-run/tasks/`.

**Критерии приемки**
- все use case покрыты;
- есть последовательность задач;
- задачи можно выполнять без домысливания;
- подход идет сверху вниз.

## Ревьюер плана

**Входы**
- `plan.md`;
- task-файлы;
- ТЗ.

**Выходы**
- `multi-agent-system/current-run/plan_review.md`.

**Критерии приемки**
- пойманы непокрытые use case;
- проверено наличие task-файлов;
- проверена формальная полнота плана.

## Разработчик

**Входы**
- один task-файл;
- код проекта;
- замечания ревьюера, если есть.

**Выходы**
- измененный код;
- тесты;
- отчет в `multi-agent-system/current-run/reports/`.

**Критерии приемки**
- реализована только поставленная задача;
- тесты запущены;
- отчет понятен и воспроизводим;
- документация обновлена при необходимости.

## Ревьюер кода

**Входы**
- task-файл;
- измененные файлы;
- тестовый отчет;
- связанная документация.

**Выходы**
- структурированное ревью по задаче.

**Критерии приемки**
- сверяет код с task-файлом;
- учитывает регрессии и тесты;
- разделяет замечания по критичности;
- не требует несвязанных улучшений.
