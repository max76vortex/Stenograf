# Acceptance Checklists

## 1. System Skeleton

- [ ] Есть `multi-agent-system/agents/` с ролями 01-09.
- [ ] Есть `multi-agent-system/` с `status.md`.
- [ ] Есть `current-run/`, `templates/`, `runbooks/`.
- [ ] Понятно, где лежат артефакты активной задачи.

## 2. Orchestrator

- [ ] Понятен вход оркестратора.
- [ ] Есть правило обновления `status.md`.
- [ ] Есть правило остановки на блокерах.
- [ ] Есть правило подтверждения этапа пользователем.

## 3. Analyst

- [ ] Аналитик генерирует полноценное ТЗ.
- [ ] Есть use case и критерии приемки.
- [ ] Есть отдельный блок открытых вопросов.

## 4. TZ Reviewer

- [ ] Review не переписывает ТЗ, а проверяет его.
- [ ] Есть разделение на `BLOCKING`, `MAJOR`, `MINOR`.
- [ ] Итоговое решение формализовано.

## 5. Architect

- [ ] Архитектура покрывает ТЗ.
- [ ] Есть модель данных и интерфейсы.
- [ ] Есть секция про развертывание и ограничения.

## 6. Architecture Reviewer

- [ ] Проверяется выполнимость.
- [ ] Проверяется совместимость с проектом.
- [ ] Замечания конкретны и ранжированы по критичности.

## 7. Planner

- [ ] Есть `plan.md`.
- [ ] Есть отдельные task-файлы.
- [ ] Есть покрытие use case.
- [ ] План идет сверху вниз.

## 8. Plan Reviewer

- [ ] Проверяется покрытие use case.
- [ ] Проверяется наличие task-файлов.
- [ ] Проверяется формальная структура плана.

## 9. Developer

- [ ] Разработчик работает по task-файлу.
- [ ] Есть тестовый отчет.
- [ ] Есть правило не выходить за задачу.

## 10. Code Reviewer

- [ ] Ревью сверяет код с task-файлом.
- [ ] Ревью смотрит на тестовый отчет и регрессию.
- [ ] Есть разделение на критичные, важные и рекомендательные замечания.

## 11. End-To-End Flow

- [ ] Вся цепочка ролей понятна.
- [ ] Есть единая рабочая зона артефактов.
- [ ] Есть операторский runbook.
- [ ] Систему можно запускать на новой задаче без переизобретения процесса.
