# Creative: Транскрибация — порядок и рабочий процесс

## Разделение фаз

- **Фаза 1 — Транскрибация (сейчас):** папка записей → порядок в исходниках → скрипт → .md в 00_inbox. Без постинга и Ghost.
- **Фаза 2 — Постинг (позже):** когда появятся материалы, готовые к публикации (статьи, кандидаты), подключаем интеграцию Obsidian ↔ Ghost и всё, что связано с постингом.

Сейчас фокус только на транскрибации: чтобы ничего не перепуталось, был явный порядок в папке с записями и в vault.

---

## 1. Папка с записями (исходники)

### 1.1. Структура папок

Рекомендуемая схема — **разбивка по месяцам**, чтобы и в проводнике, и при запуске скрипта был понятный порядок и батчи.

**Вариант A (простой):**
```
recordings/
  2024-01/
  2024-02/
  ...
  2025-03/
```
В каждой папке — все mp3 за этот месяц. Имена файлов — с датой и порядком (см. ниже).

**Вариант B (глубже по годам):**
```
recordings/
  2024/
    01/
    02/
    ...
  2025/
    03/
```
Скрипт должен уметь обходить вложенные папки (рекурсивно или по заданному уровню).

**Итог:** выбираем **Вариант A** — один уровень `YYYY-MM`. Проще бэкапить, проще гонять скрипт по одной папке за раз (`recordings/2024-03`), не смешивать месяцы.

### 1.2. Именование файлов

Чтобы не перепутать порядок и однозначно сопоставлять запись с транскриптом:

- **Обязательно:** дата + порядковый номер в день.
- **Опционально:** короткая метка (тема, место).

**Шаблон имени:** `YYYY-MM-DD_NNN[ _метка].mp3`

- `YYYY-MM-DD` — дата записи (обязательно).
- `NNN` — номер в этот день: 001, 002, … (обязательно).
- `_метка` — опционально: латиница/цифры/дефис, без пробелов в имени файла.

**Примеры:**
- `2024-03-15_001.mp3`
- `2024-03-15_002_idei-produkt.mp3`
- `2024-11-20_001_razmyshleniya.mp3`

Плюсы:
- Сортировка по имени = хронологический порядок.
- В один день несколько записей не пересекаются (разный NNN).
- Дата в имени — скрипт уверенно пишет её в frontmatter; не зависим от даты изменения файла.
- Один и тот же файл всегда даёт один и тот же .md (имя выходного файла строится из имени mp3).

### 1.3. Что делать со старыми записями (без такого имени)

- **Не переименовывать автоматически** массово без бэкапа.
- Варианты:
  1. Вручную переименовать по шаблону (хотя бы дату и номер подставить), потом прогнать скрипт.
  2. Оставить как есть; скрипт возьмёт дату из mtime и сгенерирует slug из имени — возможны коллизии, если два файла дадут один slug. Тогда лучше переименовать хотя бы в формате `YYYY-MM-DD_001_старое-имя.mp3`.

В документации (README транскрибации) явно описать: для предсказуемого порядка и отсутствия путаницы желательно использовать шаблон `YYYY-MM-DD_NNN[_метка].mp3`.

## 2. Выход транскрибации (vault 00_inbox)

### 2.1. Имена .md

Имя выходного файла — из имени mp3 (без расширения), приведённое к безопасному slug. Тогда:

- `2024-03-15_001.mp3` → `2024-03-15_001.md`
- `2024-03-15_002_idei-produkt.mp3` → `2024-03-15_002_idei-produkt.md`

Так сохраняется хронология при сортировке по имени в 00_inbox и однозначная связь «mp3 ↔ .md» по полю `audio_file` во frontmatter.

### 2.2. Один общий 00_inbox или подпапки по месяцам

- **Один плоский 00_inbox** — проще: все транскрипты в одном месте, сортировка по имени = по дате. Рекомендуется.
- Подпапки по месяцам (00_inbox/2024-03/, …) возможны, но усложняют обзор и правила скрипта; при необходимости добавить позже.

### 2.3. Связь «запись — транскрипт»

- В каждом .md во frontmatter: `audio_file: "2024-03-15_001.mp3"` (имя файла).
- Опционально позже: `source_folder: "2024-03"` или полный путь, если будем хранить записи не в одной плоской папке. Пока достаточно имени файла: при едином правиле именования запись можно найти в папке месяца.

## 3. Рабочий процесс (шаги)

1. **Подготовка записей**
   - Складывать новые mp3 в папку месяца `recordings/YYYY-MM/`.
   - Именовать по шаблону `YYYY-MM-DD_NNN[_метка].mp3`.
   - Старые при желании постепенно переименовывать (с бэкапом).

2. **Запуск транскрибации**
   - Запускать скрипт по одной папке месяца:  
     `python transcribe_to_obsidian.py "D:\1 ЗАПИСИ ГОЛОС\recordings\2024-03" "D:\Obsidian\Audio Brain\00_inbox"`
   - Или по корню `recordings` с рекурсией по подпапкам (если скрипт поддерживает).
   - Без `--overwrite` уже существующие .md не трогаем — можно безопасно повторять запуск после добавления новых файлов.

3. **После запуска**
   - В 00_inbox появляются/обновляются только .md для новых или перезаписанных (при --overwrite) записей.
   - Проверка: в Obsidian открыть 00_inbox, убедиться, что даты и имена соответствуют папке записей.

4. **Учёт «что уже переведено»**
   - Вариант 1: не вести отдельный учёт — факт наличия .md в 00_inbox с нужным `audio_file` = «уже транскрибировано». При необходимости отчёт «какие mp3 ещё без .md» можно добавить отдельной командой/скриптом.
   - Вариант 2: опциональный манифест (см. ниже) — список обработанных пар (mp3, .md) и дата запуска для логов.

---

## 4. Манифест (опционально)

Чтобы явно фиксировать, что и когда обработано:

- Скрипт поддерживает флаг `--manifest path/to/log.csv`.
- После каждого успешного преобразования mp3 → .md в конец файла дописывается строка:  
  `YYYY-MM-DDTHH:MM:SS,<путь_к_mp3>,<имя_md>,<дата_из_frontmatter>`.
- Так можно:
  - видеть историю запусков;
  - строить отчёт «какие mp3 из папки уже есть в манифесте»;
  - не хранить эту логику в самом vault.

Манифест — дополнение, не обязательное для базового порядка; можно ввести во второй итерации скрипта.

---

## 5. Чего избегать (чтобы ничего не перепутать)

- Не менять имена уже обработанных mp3 без необходимости (связь по `audio_file` сломается; при переименовании лучше перегнать транскрипт и обновить .md).
- Не класть в одну папку записей файлы с одинаковым итоговым slug (например два разных файла, дающих `2024-03-15_001.md`) — второй перезапишет первый при `--overwrite` или будет пропущен без overwrite, но тогда один mp3 останется без транскрипта.
- Транскрибацию гонять по папкам/батчам последовательно (например месяц за месяцем), а не вперемешку из разных корней без понятной схемы — так проще сопоставлять «папка записей ↔ период в 00_inbox».

## 6. Резюме решений

| Вопрос | Решение |
|--------|--------|
| Структура папки записей | `recordings/YYYY-MM/` (один уровень по месяцам). |
| Именование mp3 | `YYYY-MM-DD_NNN[ _метка].mp3` (дата + номер в день + опциональная метка). |
| Выход .md | Плоский 00_inbox; имя .md = slug от имени mp3 (сохраняем дату и номер). |
| Связь запись ↔ транскрипт | Frontmatter `audio_file` = имя mp3; при необходимости позже — путь/папка. |
| Повторный запуск | По умолчанию не перезаписывать существующие .md; при добавлении новых mp3 — только новые. |
| Манифест | Опционально, флаг `--manifest` для лога обработанных файлов. |
| Постинг / Ghost | Отдельная фаза; после появления готовых к постингу материалов. |
