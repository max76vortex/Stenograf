# Creative: полный пайплайн Audio Brain (аудио → публикация)

## Цель документа

Зафиксировать **сквозной процесс** от сырой записи до материалов, готовых к дальнейшей работе (и к фазе Ghost), включая **этап после транскрибации**: подготовка текста, адекватность, разнесение по категориям в структуре vault.

Связанные документы: `creative-transcription-workflow.md` (до `00_inbox`), `creative-content-pipeline-and-ghost.md` (Ghost), шаблоны в vault `20_system/templates/`.

---

## 1. Сквозная схема (высокий уровень)

```
[Диктофон / файлы]
        ↓
recordings/YYYY-MM/  +  имена YYYY-MM-DD_NNN[_метка].mp3
        ↓
┌───────────────────────────────────────────────────────────┐
│ Фаза A — Ingest + Транскрибация (автомат)                    │
│ mp3 переносится в asset-папку + создаётся transcript .md     │
│ затем публикация transcript в 00_inbox                        │
└───────────────────────────────────────────────────────────┘
        ↓
┌───────────────────────────────────────────────────────────┐
│ Фаза B — Подготовка и классификация (LLM/скрипты сразу)      │
│ человек подтверждает результат и корректирует при необходимости│
│ 00_inbox → правка смысла/формы → выбор типа → 10_processed/  │
│   ideas | articles | projects                               │
└───────────────────────────────────────────────────────────┘
        ↓
┌───────────────────────────────────────────────────────────┐
│ Фаза C — Развитие материала (по необходимости)              │
│ Идея → черновик статьи; проект → вехи; статья → редактура    │
└───────────────────────────────────────────────────────────┘
        ↓
┌───────────────────────────────────────────────────────────┐
│ Фаза D — Публикация (Ghost), отдельный горизонт             │
│ type: article, статусы, синк — см. creative-content-pipeline  │
└───────────────────────────────────────────────────────────┘
```

**Разделение контуров** (как в регламенте vault): сырьё и «ещё не разобрано» — `00_inbox`; осмысленно разобранное — `10_processed`; инструкции — `20_system/`.

## 1.1. Asset-папка (новая базовая единица работы)

Для каждой записи создаётся отдельная рабочая папка (asset), где живут исходник и все производные.

**Корень рабочей зоны (пример):** `D:\1 ЗАПИСИ ГОЛОС\audio-work\`

**Шаблон папки asset:**
`YYYY-MM-DD_NNN_slug__src-YYYYMMDD`

- `YYYY-MM-DD_NNN` — из имени исходного файла, если в нём есть дата+номер.
- `slug` — безопасный slug из исходного имени.
- `src-YYYYMMDD` — дата исходной записи/файла (см. приоритет ниже).

**Приоритет определения даты (`src-date`):**
1. дата из имени файла;
2. если её нет — `mtime` (в Windows это поле «Дата изменения»);
3. если недоступно и это — текущая дата.

**Что хранится внутри asset-папки:**
- исходный аудиофайл **с исходным именем** (файл переносится из `recordings`);
- `01_transcript__inbox.md` — первичный транскрипт;
- `02_clean__review.md` — очищенный текст (фаза B);
- `03_content__idea.md` / `03_content__article.md` / `03_content__project.md` — классифицированные материалы;
- `meta.json` — машинные метаданные и статусы.

Примечание: суффиксы (`__inbox`, `__review`, `__idea|article|project`) нужны для быстрого визуального понимания этапа прямо в проводнике.

## 1.2. Публикация в Obsidian как «проекция»

- В `00_inbox` отправляется сырой транскрипт (`01_transcript__inbox.md`).
- После обработки и классификации в `10_processed/ideas|articles|projects` отправляются соответствующие производные.
- Во frontmatter каждой заметки сохраняется связь с asset-папкой и исходником (`audio_file`, `asset_path`).

## 2. Фаза B — детализация (что считать «реализованным функционалом»)

Функционал здесь — **автоматизированный LLM/скриптами конвейер** с обязательным подтверждением человеком на контрольных точках.

### 2.1. Подэтапы

| Шаг | Смысл | Результат |
|-----|--------|-----------|
| **B1. Авто-триаж (LLM/скрипт)** | Скрипт оценивает базовую пригодность и тип контента | Черновой label + flag `needs_review` при рисках |
| **B2. Авто-очистка** | Нормализация ASR-текста (пунктуация, повторы, шумовые вставки) | `02_clean__review.md` |
| **B3. Авто-оценка адекватности** | Проверка на галлюцинации/ломаный язык/аномалии | score + причины в `meta.json` |
| **B4. Авто-классификация** | Модель присваивает категорию: **idea/article/project** | Суффикс имени + `category` в frontmatter/meta |
| **B5. Авто-оформление** | Применение шаблона и заполнение полей (`type`, `status`, `tags`, `audio_file`, `asset_path`) | Готовый `03_content__*.md` |
| **B6. Авто-перенос** | Скрипт публикует результат в `10_processed/...` по категории | Файл в целевой папке + ссылка на asset |
| **B7. Human checkpoint** | Человек быстро подтверждает/правит результат | Подтверждённый материал без ручного полного цикла |

### 2.2. Правила классификации (рабочая версия)

- **ideas** — короткие зарисовки, список тезисов, «на будущее», без готовой статьи.
- **articles** — есть задел под публикацию: структура, тезисы, можно развивать в черновик по шаблону статьи.
- **projects** — всё, что ведётся как направление работы (несколько шагов, контекст, сроки — по шаблону проекта).

*Итерация:* правила можно уточнять в README под `10_processed/` или в отдельной заметке-глоссарии в vault.

### 2.3. Связь с исходником

- Поле **`audio_file`** в frontmatter не удалять.
- Добавить **`asset_path`** (путь до asset-папки), чтобы быстро находить исходник и производные.
- При переносе/публикации сохранять ссылку на исходный `01_transcript__inbox.md` (поле `source_transcript` или wikilink).

### 2.4. Маркировка (имя + frontmatter + meta.json)

Маркировка должна дублироваться в трёх местах:

- **Имя файла:** суффиксы этапа/категории (`__inbox`, `__review`, `__idea`, `__article`, `__project`).
- **Frontmatter:** `type`, `status`, `tags`, `audio_file`, `asset_path`.
- **meta.json:** машиночитаемые поля (`phase`, `category`, `current_version`, timestamps, refs).

### 2.5. Что остаётся за человеком (при LLM-first процессе)

- Финальное подтверждение результата (checkpoint B7).
- Корректировка сложных кейсов, где модель неуверенна (`needs_review`).
- Полноценная литературная правка (по желанию, не блокирует базовый проход).

### 2.6. Переобработка (reprocessing)

Если asset прогоняется повторно, не затирать историю молча:

- обновлять версию (`.v2`, `.v3`) для соответствующего этапа;
- в `meta.json` фиксировать `reprocessed_at`, `reason`, `current_version`, `previous_versions`;
- публиковать в Obsidian только актуальную версию, сохраняя ссылку на предыдущую при необходимости.

### 2.7. Техническая рамка LLM/скриптов (обязательная для фазы B)

- Локальная LLM (LM Studio/Ollama) вызывается скриптом фазы B для B1-B5.
- Автоперенос (B6) разрешён только при наличии валидной категории и структуры frontmatter.
- При низкой уверенности/аномалии скрипт проставляет `needs_review` и оставляет файл в review-состоянии до подтверждения человеком.
- Метрики уверенности Whisper остаются опциональным усилением и могут быть добавлены отдельно.

## 5. Реализационный чеклист для `transcribe_to_obsidian.py` (Phase A -> asset)

Ниже конкретный список изменений в текущем `transcribe_to_obsidian.py`, чтобы фаза A сразу создавала asset-папки и переносила исходник:

1. Добавить новые CLI-аргументы:
   - `asset_root` (новый positional или `--asset-root`) — корень рабочей зоны asset.
   - `--move-source` (bool, default true для нового режима) — перенос исходника в asset-папку.
   - `--publish-inbox` (bool) — публиковать `01_transcript__inbox.md` в `00_inbox`.
   - `--source-name-mode` (default `keep-original`) — оставлять исходное имя без переименования.
2. Реализовать `build_asset_folder_name(audio_path)`:
   - дата из имени -> иначе `mtime` -> иначе now;
   - шаблон: `YYYY-MM-DD_NNN_slug__src-YYYYMMDD`.
3. На каждом файле в цикле:
   - создать asset-папку;
   - при `--move-source` перенести исходный mp3 в asset-папку;
   - генерировать транскрипт из файла уже в asset-папке.
4. Генерировать первичный markdown как:
   - файл `01_transcript__inbox.md` внутри asset-папки;
   - frontmatter: `audio_file` (исходное имя), `asset_path`, `status: inbox`, `phase: A`.
5. Публикация в Obsidian:
   - при включённом `--publish-inbox` копировать этот же markdown в `00_inbox` (как проекцию, не источник правды).
6. Добавить/обновить `meta.json` в asset-папке:
   - поля: `phase=A`, `audio_file`, `asset_path`, `src_date`, `created_at`, `updated_at`, `versions`.
7. Обновить манифест:
   - дописать колонки `asset_dir`, `source_moved`, `inbox_published`.
8. Идемпотентность:
   - если `01_transcript__inbox.md` существует и без `--overwrite`, файл пропускать;
   - исключить повторный перенос уже перемещённого исходника.
9. Обновить `check_coverage.py`:
   - добавить режим проверки не только `00_inbox`, но и `asset_root` (`01_transcript__inbox.md` по каждому аудио).
10. Обновить docs:
   - `transcription/README.md` и `SETUP.md`: новый flow с `asset_root`, примеры команд и структура папок.
