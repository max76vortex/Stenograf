# Creative: Audio → Obsidian → Ghost

## Контекст
- Большой объём аудио (≈30 ГБ mp3) с идеями, проектами и размышлениями.
- Есть отдельный Obsidian-vault под аудио и локальные LLM (LM Studio, Ollama).
- В качестве CMS выбрана Ghost в режиме «CMS с фронтом» (стандартный блог с темой).
- Обслуживание должно быть максимально дешёвым и по возможности локальным.

## Ключевые решения

### 1. Общее разделение ролей
- **Obsidian-vault**: мастер-репозиторий текста и статусов.
  - Хранит транскрипты, идеи, черновики статей, проекты.
  - В YAML/frontmatter у статей присутствуют как минимум:
    - `type: article`
    - `status: article-draft | published | archived`
    - `ghost_id` или `ghost_slug` для связи с постом в Ghost.
- **Ghost (CMS с фронтом)**:
  - Отвечает за отображение статей на сайте и публичную часть.
  - Принимает контент из Obsidian и хранит собственные статусы публикации.

### 2. Поток Obsidian → Ghost
- Точка входа: заметка в Obsidian с `type: article` и `status: article-draft`.
- Содержимое:
  - Заголовок, тело статьи, метаданные (теги, дата, slug и т.п., либо в YAML, либо выводимы из него).
- Механика:
  - Интеграция (скрипт/служба/n8n и т.п.) периодически:
    - сканирует статьи в Obsidian с `status: article-draft` (и без `ghost_id` либо с пометкой «требует синка»);
    - создаёт или обновляет посты в Ghost через Admin API;
    - при первом создании записывает `ghost_id`/`ghost_slug` обратно в frontmatter заметки.
- Статус по умолчанию в Ghost после создания:
  - либо `draft`, либо сразу `published` — определяется политикой публикации, но источник контента всегда Obsidian.

### 3. Поток Ghost → Obsidian (продвинутый сценарий)
- Цель: несколько раз в день синхронизировать статусы публикации из Ghost обратно в Obsidian.
- Механика:
  - Интеграция по расписанию (например, несколько раз в сутки):
    - обходит список статей в Obsidian с заполненным `ghost_id`/`ghost_slug`;
    - по каждому идентификатору запрашивает через API Ghost текущий статус поста (draft, scheduled, published, archived и т.п.);
    - сравнивает с текущим значением `status` в frontmatter Obsidian.
  - При отличиях:
    - обновляет поле `status` в YAML нужной заметки Obsidian в соответствии с состоянием в Ghost (например, `article-draft` → `published`).

### 4. Обязательная валидация обновления статуса в Obsidian
- После попытки обновить статус заметки в Obsidian по данным из Ghost интеграция **обязана проверить**, что изменение действительно применилось.
- Варианты реализации проверки:
  - Сразу после записи файла:
    - повторно прочитать тот же файл и убедиться, что `status` в YAML совпадает с ожидаемым значением;
  - либо (если используется git-репозиторий для vault’а):
    - проанализировать свежий diff и убедиться, что строка с `status:` изменилась на нужное значение.
- При неуспешной валидации:
  - логировать ошибку (и, при необходимости, помечать заметку специальным тегом/статусом для ручной проверки).

### 5. Частота и нагрузка
- Проверка статусов в Ghost выполняется **не постоянно**, а по расписанию:
  - несколько раз в день (например, каждые 4–6 часов), чтобы:
    - не создавать лишнюю нагрузку на API Ghost;
    - не мешать ручному процессу написания/редактирования статей в Obsidian.
- При необходимости можно ограничивать список проверяемых статей:
  - только те, у которых `status` в Obsidian ещё не `archived` и/или недавно менялся.

## Схема YAML/frontmatter для статей

### 1. Поля в Obsidian и соответствие полям Ghost

```yaml
---
type: article                  # внутренний тип в Obsidian
title: Как я пришёл к продукту X
slug: kak-ya-prishel-k-produktu-x
status: article-draft          # article-draft | ready-to-publish | published | archived
ghost_id: null                 # будет заполнен после первого синка с Ghost
ghost_status: draft            # кэш статуса из Ghost: draft | scheduled | published | archived
published_at: null             # дата фактической публикации из Ghost (ISO-формат)
tags: [личный-опыт, продукт-X] # теги в Obsidian, которые можно маппить на теги/метки Ghost
canonical_url: null            # опционально, если в Ghost задаётся канонический URL
excerpt: null                  # опциональный краткий анонс/выдержка
---
```

- `title` → маппится на `title` в Ghost.
- `slug` → маппится на `slug` в Ghost (если пустой, можно генерировать из title).
- `status` (Obsidian):
  - `article-draft` — черновик в процессе написания.
  - `ready-to-publish` — содержательно готов, ждёт публикации.
  - `published` — опубликован (по данным Ghost).
  - `archived` — снят с публикации/отправлен в архив.
- `ghost_status` — прямое отражение статуса в Ghost (`draft`, `scheduled`, `published`, `archived`), хранится отдельно от внутреннего `status`, чтобы:
  - не терять “сырой” статус из CMS;
  - иметь возможность строить правила преобразования (например, `ghost_status: published` → `status: published`).
- `published_at` — заполняется из Ghost (поле `published_at`) при первой публикации и при изменениях.

### 2. Пример полного frontmatter для опубликованной статьи

```yaml
---
type: article
title: Как я пришёл к продукту X
slug: kak-ya-prishel-k-produktu-x
status: published
ghost_id: "64f1b2c3e9f5a60001a23456"
ghost_status: published
published_at: 2026-03-15T10:23:45.000Z
tags: [личный-опыт, продукт-X]
canonical_url: https://blog.example.com/kak-ya-prishel-k-produktu-x/
excerpt: Краткое описание статьи для анонсов и предпросмотра.
---
```

### 3. Логика преобразований статуса

- При синке **Obsidian → Ghost**:
  - Если `ghost_id` пустой:
    - создаётся новый пост в Ghost;
    - Obsidian-скрипт получает `id`, `status`, `published_at` и записывает:
      - `ghost_id`, `ghost_status`, `published_at` (если есть);
    - локальное поле `status` при этом может остаться `article-draft` или стать `ready-to-publish` — по выбранной политике.
  - Если `ghost_id` задан:
    - пост обновляется в Ghost;
    - `ghost_status` и `published_at` перезаписываются актуальными значениями.

- При синке **Ghost → Obsidian**:
  - По `ghost_id` вытягивается объект поста из Ghost, читаются:
    - `status` (draft/scheduled/published/archived),
    - `published_at`.
  - Обновляются поля:
    - `ghost_status` — напрямую из Ghost;
    - `published_at` — напрямую из Ghost;
    - `status` — по простой карте правил, например:
      - `ghost_status: draft` → `status: article-draft`
      - `ghost_status: scheduled` → `status: ready-to-publish`
      - `ghost_status: published` → `status: published`
      - `ghost_status: archived` → `status: archived`

## Дальнейшие шаги
- Зафиксировать эту YAML-схему в шаблоне статьи в Obsidian.
- Выбрать конкретный механизм интеграции (локальный скрипт, служба на VDS, n8n-поток и т.п.), который:
  - умеет читать/разбирать YAML из заметок Obsidian;
  - обращается к Ghost Admin API;
  - реализует логику синка статусов и обязательной валидации записей.
- Протестировать полный цикл на 1–2 статьях:
  - создание статьи в Obsidian → появление/обновление поста в Ghost;
  - публикация/изменение статуса в Ghost → корректное обновление `ghost_status` и `status` + успешная повторная проверка содержимого файла Obsidian.
