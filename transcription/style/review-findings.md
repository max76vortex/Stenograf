# Review Findings — Phase B Quality Audit (2026-04-16)

## Обзор
Проведён ручной review 4 representative файлов из `10_processed/` (articles, ideas, projects, junk/unclear).

**Цель:** выявить типовые проблемы текущего `phase-b-process` (Kimi / Cursor skill) перед финализацией канонической инструкции.

## Примеры

### 1. Хороший (articles)
- `10_processed/articles/2017-01-26_...__article.md`
- **Плюсы:** coherent текст, хорошее резюме, структура, осмысленные tags, практическая ценность.
- **Минусы:** упоминание "шумных ASR-участков" — требует финальной ручной редакции.
- **Вывод:** такие материалы можно считать приемлемыми после лёгкой правки.

### 2. Слабый (ideas)
- `10_processed/ideas/2019-07-25_...__idea.md`
- **Проблемы:** крайне короткий текст, резюме в виде массива строк, почти нет развития мысли.
- **Вывод:** модель "сдалась" на этом ассете. Нужно усиливать prompt на генерацию содержательного контента.

### 3. Очень слабый (projects)
- `10_processed/projects/2021-05-12_...__project.md`
- **Проблемы:** практически пустой файл. Только заголовок + одна общая фраза. Не использует шаблон проекта.
- **Вывод:** для категории `project` текущий prompt недостаточно сильный. Нужно отдельное усиление промпта для этой категории.

### 4. Хороший (junk)
- `10_processed/junk/2020-05-20_...__junk.md`
- **Плюсы:** правильно определил техническую запись (автоответчик), дал понятное объяснение, почему junk.
- **Вывод:** категория `junk`/`unclear` работает хорошо.

## Типовые проблемы
1. **Недостаточная глубина** на некоторых ассетах (особенно ideas и projects).
2. **Не всегда используется полный шаблон** для категории (project особенно страдает).
3. **Резюме иногда в нечитаемом формате** (массив вместо текста).
4. `needs_review=true` ставится, но не всегда приводит к достаточному качеству даже после обработки.
5. **ASR-артефакты** (повторы, шум) не всегда хорошо очищаются автоматически.

## Рекомендации
- Усилить prompt для категорий `idea` и `project` (более конкретные инструкции по структуре и объёму).
- Добавить в `style_profile.md` и `editing_checklist.md` явные требования к минимальному объёму и структуре.
- Для `phase-b-process` skill добавить fallback: если текст слишком короткий — делать второй проход с более сильным промптом.
- В инструкции Phase B явно указать, что после B6 пользователь **обязан** проверить файлы с `needs_review=true` и категории project/idea.
