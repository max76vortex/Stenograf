# Старт мультиагентного прогона (Variant B)

Исполняй эту процедуру **строго по порядку**. Опирайся на runbook: `multi-agent-system/runbooks/how-to-run-project-through-multi-agent-system.md`.

---

## Фаза A — Preflight и подготовка рабочего пространства

**Сначала** убедись, что окружение и репозиторий готовы к запуску оркестратора. **Не переходи к фазе B**, пока критичные пункты preflight не закрыты (или пользователь явно принимает риск в письменном виде).

### A1. Автоматическая проверка (рекомендуется)

Предложи пользователю выполнить **из корня workspace** (`N8N-projects`):

```powershell
pwsh -NoLogo -File "multi-agent-system/tools/orchestrator-preflight.ps1"
```

- Если **exit code 1** или есть `[FAIL]` — остановись, помоги устранить проблемы по `multi-agent-system/runbooks/prerequisites.md`, **не переходи к фазе B**.

### A2. Ручная проверка тем же агентом (если терминал недоступен или для дубля)

Проверь сам (чтение/логика):

1. **PowerShell 7:** запуск оркестратора в документации идёт через `pwsh`; без него пользователь должен знать альтернативу.
2. **Cursor Agent CLI:** хотя бы один рабочий путь к вызову агента:
   - `Get-Command agent` / `cursor-agent`, **или**
   - файлы `agent.cmd` / `cursor-agent.cmd` в `%LOCALAPPDATA%\cursor-agent\`, **или**
   - крайний случай — `cursor` в PATH (как в `run-agent.ps1`).
3. **Обязательные файлы** (относительно корня workspace):
   - `multi-agent-system/agents/01_orchestrator.md`
   - `multi-agent-system/start-orchestrator.ps1`
   - `multi-agent-system/run-agent.ps1`
   - `multi-agent-system/status.md`
4. **Опционально (предупреждение, не блокер до фазы B):** отсутствуют `current-run/task_brief.md` или `project_context.md` — их заполнишь в фазе B.

### A3. Состояние прогона (п.2 runbook: run готов)

1. Прочитай `multi-agent-system/status.md` и кратко опиши пользователю:
   - `Status`, `Current stage`, `Active task`;
   - есть ли незавершённый прогон (не `COMPLETE`).
2. Просмотри `multi-agent-system/current-run/` (наличие старых артефактов, `README.md`).
3. **Спроси пользователя одним блоком:**
   - продолжаем текущий/последний run **или** начинаем **новую** задачу?
4. Если **новая задача** — напомни правило из `multi-agent-system/current-run/README.md`: нужно очистить или заархивировать результаты прошлого прогона. **Не удаляй файлы без явного согласия пользователя.** Предложи варианты: ручное копирование в архив, или перечисли что именно пользователь должен удалить/перенести.
5. Зафиксируй решение пользователя в ответе (1–2 предложения), чтобы дальше не путать контексты.

**Выход фазы A:** preflight пройден (или явно принят риск), понятно, можно ли безопасно писать новые `task_brief.md` / `project_context.md` и стартовать оркестратор.

---

## Фаза B — Входы перед стартом (п.1)

Собери ответы **вопросами к пользователю** (можно одним сообщением со списком). Цель — заполнить два файла:

- `multi-agent-system/current-run/task_brief.md`
- `multi-agent-system/current-run/project_context.md`

### Вопросы для task brief (что сделать)

- Краткое название / тема задачи.
- Цель: что должно быть на выходе (1–3 предложения).
- Критерии успеха / приёмки (что считаем «готово»).
- Scope: что **входит** и что **явно не входит**.
- Ограничения: сроки, стек, инфраструктура, безопасность, «нельзя трогать».
- Есть ли ссылки на существующие документы, issue, заметки — перечислить пути.

### Вопросы для project context (контекст проекта)

- **MAS Project ID:** уникальный slug (латиница, дефисы) для архива — например `my-feature-v1`; будет использоваться в `archive/mas-projects/<id>/` и в Obsidian.
- Название проекта и краткое описание продукта/модуля.
- Стек (языки, фреймворки, ключевые сервисы).
- Структура репозитория: важные папки и файлы (пути от корня workspace).
- Текущее состояние: что уже сделано, что сломано, что является источником правды.
- Как запускать локально / тесты (если известно).
- Любые блокеры до старта мультиагентного цикла.

После получения ответов **создай или перезапиши** оба файла цельным, человекочитаемым markdown. Язык — как у пользователя в ответах (по умолчанию русский). В `project_context.md` обязательно включи строку `**MAS Project ID:** \`<slug>\`` для будущей обязательной архивации (см. `runbooks/mas-archive-policy.md`).

**Выход фазы B:** файлы `task_brief.md` и `project_context.md` существуют, в контексте задан slug, согласованы с пользователем по смыслу.

## Фаза C — Подтверждение перед запуском

1. Кратко перечисли, что будет запущено:
   - путь к `task_brief.md` и `project_context.md`;
   - что старт идёт через оркестратор (Variant B, checkpoints).
2. **Явно спроси:** «Подтверждаете запуск оркестратора?» (ожидай явного «да» / «запуск» / эквивалента).

**Не переходи к фазе D без этого подтверждения.**

---

## Фаза D — Запуск через оркестратор (п.3)

Только после подтверждения:

1. При необходимости ещё раз напомни: `multi-agent-system/runbooks/prerequisites.md`.
2. Дай пользователю **готовую команду** для выполнения **из корня workspace** (`N8N-projects`).

По умолчанию `start-orchestrator.ps1` уже читает `multi-agent-system/current-run/task_brief.md` и `project_context.md` (если файл существует):

```powershell
.\multi-agent-system\start-orchestrator.ps1 -Model "auto"
```

Если нужны нестандартные пути к постановке/контексту:

```powershell
.\multi-agent-system\start-orchestrator.ps1 `
  -TaskFile "multi-agent-system/current-run/task_brief.md" `
  -ProjectContextFile "multi-agent-system/current-run/project_context.md" `
  -Model "auto"
```

(Замени `auto` на нужную модель, если пользователь указал другую.)

3. Кратко опиши, чего ожидать дальше: первый checkpoint, смотреть `multi-agent-system/status.md`, артефакты в `current-run/`, подтверждать этапы по runbook.

**Не подменяй** запуск оркестратора обычным «давай сделаем в чате без статуса и артефактов», если пользователь выбрал эту команду.

---

## Если что-то пошло не так

- Блокеры — через `multi-agent-system/current-run/open_questions.md` и обновление `status.md` по правилам оркестратора.
- Не перескакивай стадии вручную.
